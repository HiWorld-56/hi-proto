# 优化目标
我希望将代码进行优化：
1. 参考 Google 的命名规则，将数据结构与rpc命名规范化。
2. 将命名“母语”化。我的母语是中文，所以在进行英文命名时，选择的单词语义可能有偏差，而，该项目可能会分享给以英语为主要交流语言的同事或合作伙伴，我希望工程尽可能专业。
3. 评估各个模块的划分是否合理，我为了将数据结构尽可能缩短，可能在模块的划分上存在问题，比如，在即时通讯服务中，group 和 user 应该是平级，但，由于两个代码块中都可能出现 Unit 这样的结构定义，所以，group被单独划分成了一个子模块，综合来看似乎并不合理。
4. 目前主要是优化proto的部分，关于http接口生成文件、makefile等，在整理完proto之后再优化。

# 数据解释
## Unit → UserSummary
我认为不应该这么改，因为这里的Unit泛指一切可加入did的对象，包括人、AI对象、IOT托管对象等，但是 user 似乎更多指人。

## q3 → relay
- 一种特殊单聊方式
- 两个单聊者可以各自拉入一个自己的机器人，机器人会阅读对方发出的信息：
  - 开启托管，机器人会代替主人自动回复
  - 关闭托管：机器人会先将生成的回复内容发给主人review，主人确认之后才发送。
- 在前端页面，永远看到是两个单聊者在聊天。
- 背后的实现原理是在通用message的基础上，在前端对群聊显示效果进行一定的控制。
综上，这只是对特殊业务等一个命名，没有特殊含义，而且以后可能出现q4、q5等，实现原理，基本都是基于通用message，在显示端做一些特殊处理。

## bot
bot换成agent是正确的，bot更偏向带运动机构的执行机器人，agent确实更像人的“代理”。

## UserExtend
UserExtend是User的扩展，是商户级用户才会有的概念。
当普通用户注册为商户级用户后，did后台会针对该用户生成：
1. 名为 DBExtend_<商户did> 的数据表，用来存储商户需要针对其用户保存的额外信息。
2. 商户扩展操作 token，商户后台调用hidid后台操作DBExtend_<商户did>数据表的接口时，用于鉴权。
即，DBExtend_<商户did> 实际上为商户提供了一个基础、可扩展的用户系统，商户可以把普通did用户拉入它的 DBExtend_<商户did> 数据表中，以便存储一些did基础用户数据结构中没有的字段，如，等级、主页、宣传卡片等。
