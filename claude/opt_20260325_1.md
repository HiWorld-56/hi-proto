## 数据解释
### Unit → UserSummary
我认为不应该这么改，因为这里的Unit泛指一切可加入did的对象，包括人、AI对象、IOT托管对象等，但是 user 似乎更多指人。

### q3 → relay
- 一种特殊单聊方式
- 两个单聊者可以各自拉入一个自己的机器人，机器人会阅读对方发出的信息：
  - 开启托管，机器人会代替主人自动回复
  - 关闭托管：机器人会先将生成的回复内容发给主人review，主人确认之后才发送。
- 在前端页面，永远看到是两个单聊者在聊天。
- 背后的实现原理是在通用message的基础上，在前端对群聊显示效果进行一定的控制。
综上，这只是对特殊业务等一个命名，没有特殊含义，而且以后可能出现q4、q5等，实现原理，基本都是基于通用message，在显示端做一些特殊处理。

### bot
bot换成agent是正确的，bot更偏向带运动机构的执行机器人，agent确实更像人的“代理”。

### UserExtend
UserExtend是User的扩展，是商户级用户才会有的概念。
当普通用户注册为商户级用户后，did后台会针对该用户生成：
1. 名为 DBExtend_<商户did> 的数据表，用来存储商户需要针对其用户保存的额外信息。
2. 商户扩展操作 token，商户后台调用hidid后台操作DBExtend_<商户did>数据表的接口时，用于鉴权。
即，DBExtend_<商户did> 实际上为商户提供了一个基础、可扩展的用户系统，商户可以把普通did用户拉入它的 DBExtend_<商户did> 数据表中，以便存储一些did基础用户数据结构中没有的字段，如，等级、主页、宣传卡片等。

## 字段id矫正
message的字段id需要矫正：
- 常规字段应该从1开始顺序命名
- 某些特殊情况，如 UpdateAssetsReq 中:
  ```
  message UpdateAssetsReq {
    string did        = 1;
    string btc        = 2;
    string eth        = 31;
    string usdt_erc20 = 32;
    string trx        = 41;
    string usdt_trc20 = 42;
    string whds_trc20 = 43;
    string bt_trc20   = 44;
    string sol        = 51;
    string usdt_sol   = 52;
    string bt_sol     = 53;
    string panda_sol  = 54;
    string apt        = 61;
    string whds_apt   = 62;
    string hwhd_apt   = 63;
    string slkj_apt   = 64;
    string wsm_apt    = 65;
  }
  ```
  双位数的字段id是为了在后续增加币种时留出扩展性，方便将同一链上的币种分配到同一段id中，以便扩展性。
### 补充
Q3GroupEventReq field 1 并非预留，InviteCodeCreateResp 的field 2 也非预留，总结：单位数的id都不是预留的，只有双位数的id才是预留的。关于双位数id，依然是从x1开始顺序向后，比如11/12/13，不应出现没有14的情况下直接跳到15，但是会出现没有14-19的情况下，直接跳到21，这里是将1x和2x分为两段id，比如，UpdateAssetsReq中，3x给了eth链，4x给了波场链。
